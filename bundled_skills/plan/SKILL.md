# Plan

작업 계획 수립을 위한 5단계 읽기 전용 워크플로우.

**이 스킬은 읽기 전용이다. 코드를 수정하지 말 것. 계획만 수립할 것.**

## 1단계: 재진술

1. 사용자의 요청을 구체적 목표로 재진술할 것
2. "성공"의 기준을 명확히 정의할 것
3. 불명확한 부분은 사용자에게 질문할 것

## 도구 선택 기준

대상이 **스킬**인 경우 워크스페이스 도구 대신 스킬 도구를 사용할 것:
- `read_file` → `read_skill` / `list_files` → `read_skill`로 스크립트 목록 확인 / `search` → `read_skill`로 내용 검색

## 2단계: 범위 분석

1. `list_files`으로 프로젝트 구조를 파악할 것 (스킬 대상이면 `read_skill`로 확인)
2. `search`으로 관련 코드를 찾을 것
3. `read_file`로 핵심 파일을 읽고 분석할 것
4. 영향 범위를 파악할 것:
   - 수정이 필요한 파일 목록
   - 영향받는 기존 기능
   - 외부 의존성

## 3단계: 위험 평가

1. 기술적 위험 요소를 식별할 것:
   - 호환성 문제
   - 성능 영향
   - 보안 고려사항
2. 불확실한 부분을 명시할 것
3. 각 위험에 대한 완화 방안을 제시할 것

## 4단계: 단계별 계획

1. 작업을 독립적인 단계로 분해할 것
2. 각 단계에 포함할 것:
   - **목표**: 이 단계에서 달성할 것
   - **대상 파일**: 수정/생성할 파일
   - **변경 내용**: 구체적인 작업 내용
   - **검증 방법**: 이 단계의 성공 확인 방법
3. 단계 간 의존성을 명시할 것
4. 예상되는 전체 작업량을 제시할 것

## 5단계: 승인 요청

1. 전체 계획을 구조적으로 정리하여 사용자에게 제시할 것
2. 대안이 있으면 장단점과 함께 제시할 것
3. 사용자의 피드백을 받아 계획을 조정할 것
4. **이 스킬은 여기서 종료.** 직접 구현을 시작하지 말 것

## 완료 후 전환 가이드

**plan은 유일하게 사용자 승인이 필요한 워크플로우다.** 계획을 제시하고 승인을 받을 것.

사용자가 승인하면, 작업 규모에 따라 다음 단계를 사용자에게 제안할 것:

- **수정 파일 2개 이하** → "구현을 시작할까요?" (impl 워크플로우로 전환)
- **수정 파일 3개 이상** → "대규모 작업이므로 coding-pipeline으로 진행할까요?" (coding-pipeline 워크플로우로 전환)
